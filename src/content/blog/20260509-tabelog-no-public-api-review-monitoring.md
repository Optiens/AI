## 食べログ API は「もう存在しない」

中小飲食店のオーナーから「**食べログの新着口コミを自動でキャッチしたい**」というご相談をいただくことがあります。低評価レビューが投稿されたら即対応したい、高評価には返信を出したい、というニーズです。

しかし結論から言うと、**食べログには公式 API が現在提供されていません**。

- 2010 年代前半に存在した食べログ API は **2014 年 6 月 30 日に提供終了**
- 以降、サードパーティ向けの公開 API は存在しない
- 利用規約・robots.txt で **スクレイピング（自動取得）は明示的に禁止**

「じゃあ無理なのか」と聞かれると、そうではありません。**店舗オーナー本人** に焦点を絞れば、合法的に新着口コミを取得する方法はあります。

---

## 合法的な代替方式 3 選

### 方式 A: 店舗管理画面の通知メールを IMAP で受信→AI で解析（推奨）

食べログの **店舗向け管理画面** には、新着口コミが投稿された際にメール通知を送る機能があります（オーナー会員契約が前提）。

仕組み:
1. 通知メールを **専用の受信ボックス** に転送（Google Workspace 等）
2. IMAP プロトコルで定期的にメールを取得
3. AI（Claude Sonnet 4.6 や GPT-4o）でメール本文から口コミ内容を抽出
4. データベースに保存し、ダッシュボードで一元管理

メリット:
- **店舗オーナー本人** の管理メールを使うため、第三者スクレイピングではない
- 食べログ規約に抵触しない
- 既存契約の範囲で実装可能

デメリット:
- 食べログ側のメール仕様変更で影響を受ける可能性
- リアルタイムではない（数分〜1 時間のラグ）

### 方式 B: 月次の手動 CSV エクスポート → AI で一括分析

食べログ管理画面から月次で口コミデータを CSV エクスポートし、AI に分析させる方式です。

メリット:
- リアルタイム性は不要・月次振り返り用途に十分
- 設定がシンプル

デメリット:
- リアルタイムでの即時返信ができない
- 手動運用なので忘れがち

### 方式 C: 予約管理 SaaS 経由のパートナー連携

トレタ・ぐるなびポケパラ等の予約管理 SaaS には、食べログ口コミとの連携機能を持つものがあります。これらを経由して取得する方式です。

メリット:
- 既存の SaaS 契約で完結
- ダブルブッキング対策などとセットで運用可能

デメリット:
- パートナー契約・追加コストが必要
- 自由度が低い

## Google マップ・Instagram は公式 API で OK

食べログ以外のチャネルは **公式 API が利用可能** です。

### Google マップ（Google ビジネスプロフィール）

- **Google Business Profile API**（公式）で自社店舗の口コミを取得・返信可能
- 無料（リクエスト上限内）
- ただし API アクセス申請の承認制（公式非公表で数日〜数ヶ月の幅。**60 日以上アクティブな確認済プロフィール**が前提）

### Instagram コメント・メンション

- **Instagram Graph API**（Facebook Business 経由）で取得・返信可能
- 無料（基本利用）
- 実運用には Meta の **App Review 承認**（2〜4 週間）が必要

これらは **規約違反の心配なく** 構築できます。

## 推奨される統合構成

中小飲食店向けの現実的な構成は以下です。

```
[食べログ通知メール] ──→ [IMAP 受信] ──→ [AI 解析]
                                            ↓
[Google ビジネスプロフィール API] ───→ [統合 DB] ──→ [ダッシュボード]
                                            ↑       └→ [LINE/Slack 通知]
[Instagram Graph API] ──────────────────────┘
```

このハイブリッド構成で、**3 チャネル横断の 24h 自動監視** が実現できます。新着口コミが投稿されたら数分以内にオーナーの LINE に通知、低評価は即時アラート、高評価は AI が返信下書きを生成、という運用が可能です。

## やってはいけないこと

- **食べログのスクレイピング（自動取得）**: 規約違反 + 法的リスク
- **第三者ツールでの無断取得**: 利用規約に違反する SaaS を使うと、店舗側も責任を問われる可能性
- **顧客個人情報の独自収集**: 個人情報保護法上の問題

「**店舗オーナー本人の管理画面通知メール**」と「**公式 API**」の範囲内で構築するのが、長く安全に運用できる方法です。
