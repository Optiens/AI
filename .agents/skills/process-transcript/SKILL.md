# 文字起こし処理フロー（記録 → 報告 → 記事化）

CEO から動画・対談・ウェビナー等の文字起こしテキストが共有された際に必ず実行する標準フロー。
情報記録、CEO への報告、SEO/ブランディング判定、元ソース匿名化、ブログ記事化、アイキャッチ生成、作成後ファクトチェック、コミット・プッシュまで一貫して進める。

**重要:** 文字起こし投入は、このフロー実行の明示指示とみなす。CEO が「記事化しない」「報告だけ」と明示した場合を除き、記事化候補の確認待ちで停止しない。

## 起動条件

CEO が以下のような形で文字起こしテキストを共有したとき、本スキルを自動的に発動する:
- 「以下は動画の文字起こしです」「文字起こしを共有します」等の前置き
- 数百〜数千語の対談・講演・インタビュー風のテキストの貼り付け
- 文字起こし共有後に「Optiens に有用な内容を記録」「記事化」の指示があるとき

CEO から明示的な指示がなくても、文字起こしであることが文脈から明らかなら本フローを開始する。

---

## Step 1: 文字起こし内容の整理・分析

### 1-1. 全体把握
- 発信者・登場人物・テーマ・日付（推定可なら）を把握する
- 全体の論旨を 100〜200 字で要約する

### 1-2. Optiens への有益度と記事化可否を評価する観点
以下の観点で「Optiens にとって有益な情報」を抽出し、SEO またはブランディングに寄与する内容は記事化候補にする。

| 観点 | 抽出対象 |
|---|---|
| **市場・業界トレンド** | AI 業界の最新動向、競合の動き、技術ロードマップ |
| **顧客理解** | 中小事業者の課題・悩み・判断軸 |
| **マーケティング・集客** | 集客導線、ブランディング、コミュニケーション設計 |
| **サービス設計** | 新しいサービスメニュー、料金構造、提供方法のヒント |
| **経営判断** | 採用・組織・財務・戦略への示唆 |
| **コンテンツ素材** | 引用できる発言、独自フレームワーク、数値・統計 |
| **リスク・注意点** | 採用すべきでない手法、避けるべき表現、規制動向 |

記事化の判定基準:
- SEO: 検索されやすい固有名詞、業務課題、ツール名、比較軸、導入方法、失敗例がある
- ブランディング: Optiens の実務性、誠実なリスク説明、AI支援事業の専門性、地域中小企業への翻訳力を示せる
- 見送り: 出典が弱い、煽りが強い、Optiens の方針に合わない、ファクトチェック不能、既存記事と重複が大きい

### 1-3. 取り入れない要素の判別

Optiens のブランドトーン（落ち着き・誠実・実務）と相反する要素は、**構造のみ借用し表現は採用しない**:
- 煽り語彙（「絶対」「人生変わる」「いま動かないと」）
- 確約表現（「必ず成功する」「○○日で稼げる」）
- 不安喚起マーケ（焦らせて判断を急がせる）
- 過剰な権威付け

## Step 2: ナレッジへの記録

### 2-1. 記録種別の判断

| 観点 | 記録タイプ | 例 |
|---|---|---|
| 業界の動向・他社事例・動画の参照 | reference | 「○○氏の集客手法分析」「AI 業界 ○年 △月の発言整理」 |
| Optiens の方針決定に関わる気づき | feedback | 「○○トーンは採用しない」「△△を必ず実施する」 |
| 進行中プロジェクトへの影響 | project | 「□□戦略を Phase A2 で開始」 |

### 2-2. ナレッジファイル作成

`executive/data/transcript-insights/` 配下に Markdown を作成し、コミット対象に含める。

ファイル名規則:
- `YYYYMMDD_reference_<英語スラッグ>.md`
- `YYYYMMDD_feedback_<英語スラッグ>.md`
- `YYYYMMDD_project_<英語スラッグ>.md`

冒頭にフロントマター:
```markdown
---
name: 日本語の名前
description: 1 行で内容を表す説明
type: reference | feedback | project
---
```

本文の構成:
- 出典（誰の発言か・動画タイトル・日付）
- 中核メッセージ（200 字以内）
- Optiens への適用ポイント（具体的に）
- 採用しない要素（あれば）
- 記事化候補（あれば）

### 2-3. Codex永続メモリへの反映が必要な場合

Codex の永続メモリに残す必要がある方針・再発防止ルールは、直接 MEMORY.md を編集しない。
ユーザーの明示指示がある場合のみ、`C:\Users\blueb\.codex\memories\extensions\ad_hoc\notes\` に追記ノートを作成する。

## Step 4: ブログ記事化（承認待ちなしで自走）

### 4-1. fact-check-content → deidentify-transcript-source → write-blog-article の順で呼び出す

SEO またはブランディングに寄与する記事化候補ごとに、先に `.agents/skills/fact-check-content/SKILL.md` を必ず実行し、その後 `.agents/skills/write-blog-article/SKILL.md` の手順に従って記事を書く。
特に以下を遵守:
- 文字起こしを事実源として扱わない。固有名詞、製品名、日付、提供範囲、数値、料金、機能名は一次情報で確認する
- `.agents/skills/deidentify-transcript-source/SKILL.md` を必ず実行し、元ソースが特定されないように構成・例・言い回しを再構成する
- 新規記事・ニュース解説・外部サービス解説では `executive/data/YYYYMMDD_<slug>_factcheck.md` にファクトチェック記録を残す
- `src/content/blog/_project-status.md` の禁止表現リストを確認
- カテゴリは `AI / business / technology / agriculture / welfare / social / announcement` のいずれか
- 数値・固有名詞は出典明記または範囲表現
- 煽り語彙・確約表現・確証バイアスを誘発する表現は使わない

#### 文字起こしから記事化する際の追加ルール（必須）

**動画と全く同じ流れにしない。元ソースが第三者に分からない程度まで再構成する。**
詳細: `.agents/skills/deidentify-transcript-source/SKILL.md` を必ず参照。

7 つの必須再構成:
1. **構成・順序を変える** ── 動画の項目順をそのままなぞらない
2. **用語・表現を Optiens 文体に** ── カタカナ→英表記、砕けた表現排除
3. **Optiens 独自の補足を加える** ── 中小事業者観点・ローカル LLM 選択肢等
4. **冗長部分を削除** ── 30 分動画 → 2,500〜3,500 字の濃縮版
5. **出典の扱い** ── 動画タイトル・YouTuber 名を本文に書かない
6. **固有例の置換** ── 元動画だけに出るデモ例・比喩・宣伝要素は、Optiens読者向けの別例に置き換える
7. **検索耐性確認** ── 特徴的な一文を検索して元動画が特定されそうなら書き換える

書き終わったら自己チェック:
- [ ] 動画タイトル・YouTuber 名が本文に出ていない
- [ ] 「動画によると」のような曖昧引用なし
- [ ] 動画の項目順をなぞっていない
- [ ] 元動画固有のデモ例・宣伝CTA・口癖が残っていない
- [ ] 7語以上連続して文字起こしと一致する文がない
- [ ] 既存 Optiens ブログとの関連が張られている
- [ ] Optiens 独自の補足が 1 つ以上ある

### 4-2. アイキャッチ画像生成（必須・スキップ禁止）

記事 md を作成したら、**同セッション内で必ず** アイキャッチ画像を生成する。
write-blog-article skill の Step 4 と同じ手順:

```bash
node scripts/generate-blog-imagen.mjs "英語プロンプト" "public/images/blog/<記事ファイル名と同一スラッグ>.webp"
```

**パス命名ルール**（過去事故の再発防止）:
- 画像ファイル名は **記事 md のファイル名（拡張子除く）と完全一致** させる
- frontmatter `image:` も同じパスにする
- 日付プレフィックスを片方だけ落とす運用は厳禁

- モデル: Google Vertex AI の Imagen Ultra 最新安定版を使う。2026-05-15 時点の公式モデル ID は `imagen-4.0-ultra-generate-001`
- 実行前に `scripts/generate-blog-imagen.mjs` の `IMAGEN_MODEL` / 既定モデルを確認し、公式ドキュメント上でより新しい安定版 Imagen Ultra が出ていればそのモデルを使う
- `GCP_PROJECT_ID` と `GOOGLE_APPLICATION_CREDENTIALS` を `.env` から自動読込
- Vertex AI / Google Cloud クレジット / 認証で失敗した場合のみ `scripts/generate-blog-gemini.mjs`（Gemini 2.5 Flash Image）を検討する
- `scripts/generate-blog-openai.mjs` は旧gpt-image-2運用のレガシー。明示指示がない限りブログアイキャッチの正規手順には使わない
- 後回しにしない（後でまとめてやろうとすると忘れる事故が起きる）
- 失敗時は CEO に報告、必要なら md の `image:` フィールドを省略

### 4-2b. 作成後ファクトチェック（必須・スキップ禁止）

記事本文とアイキャッチ生成後、公開前に `.agents/skills/fact-check-content/SKILL.md` を再実行する。

確認対象:
- 本文の固有名詞、数値、日付、料金、提供範囲、将来予測、比較表現
- frontmatter の `title` / `excerpt` / `date` / `image`
- 画像ファイルの存在、記事 slug と画像 slug の一致
- 出典URLとファクトチェック記録の存在
- 文字起こし由来の未確認断定が残っていないこと
- deidentify-transcript-source の公開前チェックリストを満たしていること
- AI活用診断・詳細レポート・導入支援・スポット相談の記述が `executive/ai-consulting/blog-service-claim-rules.md` と一致していること

公開前にサービス範囲表現チェックも必ず実行する:

```bash
npm run check:blog-service-claims
```

エラーが出た記事は公開・コミットしない。警告が残る場合は、本文を修正するか、正本に基づいて許容する理由をファクトチェック記録に残す。

### 4-3. ブログ候補リストへの追記

`executive/data/blog-candidates.md` に該当記事を ✅ 付きで追加する（記事化前なら未マークで追加）。ファイルがなければ作成する。

### 4-4. コミット・プッシュ

可逆な記事追加は CEO の承認待ちなしで `git add → commit → push origin main` まで実行する。

**画像ファイル（`public/images/blog/<slug>.webp`）も同コミットに含める**。
**ファクトチェック記録（`executive/data/YYYYMMDD_<slug>_factcheck.md`）も同コミットに含める**。

コミットメッセージ例:
```
feat(blog): 文字起こし分析から記事 N 本を執筆

- ファイル名 (#候補番号): 1 行要約
- ファイル名 (#候補番号): 1 行要約

出典: <動画/対談タイトル等>

Co-Authored-By: Codex Opus 4.7 (1M context) <noreply@anthropic.com>
```

## 例外・注意事項

### 文字起こしが部分的・不明瞭な場合
全体の論旨が読み取れない場合でも、読み取れる範囲で有益情報の記録と報告は行う。記事化に必要な事実確認ができない場合のみ、ブログ化は保留し、必要な追加情報を CEO に確認する。

### 機密情報を含む場合
文字起こしに顧客名・個人情報・取引条件等の機密情報が含まれる場合は、ナレッジ記録時に**マスキングまたは抽象化**してから保存する。

### 競合事業者を直接批判するトーンになりそうな場合
他社・他者を否定的に扱う記事は Optiens のブランドトーンと相反する。「採用しない要素」として整理し、肯定的な学びだけを記事化する。

### Optiens の事業段階（Phase A1）と相反する記事
「導入実績がある」「○○件支援した」等の実績語が文字起こし元の文脈にあったとしても、Optiens の記事では使えない。**Optiens 自身の記事に翻訳する際に必ず弱化する**。
