# 公開コンテンツ ファクトチェック必須スキル

Optiens の公開コンテンツを作成・修正・公開するときは、本文作成後ではなく、**公開前の必須ゲート**として本スキルを実行する。

## 起動条件

次のいずれかに該当する場合は必ず使う。

- ブログ、SNS投稿、ニュース解説、動画文字起こし記事、LP、サービスページ、FAQ、スライド、レポートを作る・直す
- 価格、返金、支払い、納期、提供範囲、MTG有無、保守条件、API仕様、外部サービス名、製品名、法令・補助金・統計・市場データに触れる
- 「最新」「本日」「昨日」「今年」「現在」「リリース」「利用可能」「準備中」「受付中」など、変わり得る状態を書く
- ユーザーが「ファクトチェック」「事実確認」「裏取り」「本当か確認」と言った

## 必須ルール

- 現在性のある外部事実は必ずWebで確認する。一次情報を優先し、公式発表、公式ドキュメント、行政資料、企業IR、原論文を使う。
- OpenAI、Google、Supabase、Vercel、freee など製品・API・料金・提供条件は、公式情報で確認する。
- 社内事実は `AGENTS.md`、`src/content/blog/_project-status.md`、関連する `executive/` 配下の正本、実装ファイルを照合する。
- 未確認の数値・比較・固有名詞・将来予定を断定しない。確認できない場合は「未確認」「可能性」「予定」「提供条件あり」などに弱めるか、削除する。
- 文字起こし由来の記事は、文字起こしを事実源として扱わない。固有名詞、機能名、日付、提供範囲、数値は必ず別途確認する。
- 文字起こし由来の記事では、記事本文・frontmatter・アイキャッチ画像作成後に本スキルを再実行し、作成後ファクトチェックを記録する。
- 「必ず」「確定」「誰でもすぐ」「完全に」「外部共有されない」「無制限」などの強い表現は、根拠がない限り避ける。

## 手順

1. **主張を抽出する**
   - 外部事実、社内事実、価格・提供条件、日付、数値、比較、将来予測、推奨の根拠に分ける。
2. **社内正本と照合する**
   - AGENTS.md の禁止事項、ブランドカラー、ロードマップ、現在フェーズに反していないか確認する。
   - ブログでは `src/content/blog/_project-status.md` も確認する。
3. **外部事実を確認する**
   - 公式・一次情報を優先する。二次記事を使う場合は、一次情報で補強する。
   - 公開日・対象地域・プラン・提供条件・プレビュー表記を確認する。
4. **判定する**
   - `OK`: そのまま掲載可
   - `要修正`: 表現を弱める、条件を加える、出典を付ける
   - `削除`: 根拠が弱い、誤解を招く、社内方針に反する
   - `保留`: 現時点で確認不能。公開しないかユーザーに確認する
5. **記録する**
   - 新規記事、ニュース解説、文字起こし由来記事、重要ページ修正では `executive/data/YYYYMMDD_<slug>_factcheck.md` を作成する。
   - 軽微な文言修正でも、最終報告に「何を確認したか」を明記する。
6. **公開前検証を行う**
   - `rg` で危険表現や旧表現が残っていないか確認する。
   - MarkdownやAstroを変更した場合は `git diff --check` と `npm run build` を実行する。

## 記録テンプレート

```markdown
# <タイトル> ファクトチェック記録（YYYY-MM-DD）

## 対象
- 対象ファイルまたは投稿予定文

## 判定
- 公開可 / 修正後公開可 / 保留 / 公開不可

## 確認結果

| 項目 | 判定 | 根拠・修正方針 |
|---|---|---|
| <主張> | OK / 要修正 / 削除 / 保留 | <出典、確認内容、修正方針> |

## 公開文で避ける表現
- <危険な断定>

## 参照元
- <公式URLまたは社内正本>
```

## 完了条件

- 重要な事実主張がすべて `OK` または `要修正後OK` になっている。
- 出典が本文またはファクトチェック記録に残っている。
- 未確認の断定が本文に残っていない。
- 最終報告で、ファクトチェック実施済みであること、確認元、修正した危険表現、実行した検証コマンドを簡潔に報告する。
