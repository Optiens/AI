# AIコーディング時代のリファクタリングと技術的負債

## 中核メッセージ

AIコーディングは実装速度を上げる一方で、人間とAIの変更が混ざるほどコードベースの一貫性が崩れやすい。リファクタリングは「きれいにする」目的だけで無限に続けるのではなく、次の変更を安全に入れる、重複処理を減らす、仕様の複雑さを下げるなど、明確な目的を決めて行う必要がある。

## Optiensへの適用ポイント

- AIで作った業務アプリの納品時は、動作確認だけでなく「次の変更が安全にできるか」を確認項目に入れる。
- リファクタリング依頼時は、AIに丸投げせず、先に人間側が到達したい設計を定義する。
- 「改善して」ではなく、「重複したDB問い合わせを1箇所に寄せる」「エラー処理の責務を分ける」「UIと業務ロジックを分離する」のように完了条件を与える。
- チームでAIコーディングを使う場合は、リポジトリ標準、命名規則、テストコマンド、レビュー観点をファイル化する。
- AI生成コードにも人間の責任者を置き、最終承認者を明確にする。
- 有償支援では、短期の実装速度だけでなく、保守・改修・引き継ぎまで含めた提案が差別化になる。
