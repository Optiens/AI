## 中核メッセージ

ローカルLLMや小規模モデルで長い資料を扱う場合、単に本文を貼り付けるだけでは、指示や重要条件が入力全体の中で薄れることがある。業務用途では、モデル・テンプレート・コンテキスト長を確認したうえで、固定ルール、参照資料、最終指示の置き方を設計し、出力品質を小さな評価セットで検証する必要がある。

## Optiensへの適用ポイント

- **ローカルAI導入支援の説明材料**: 機密資料を扱う顧客に対し、ローカルLLMは「置けば終わり」ではなく、プロンプト配置・テンプレート・評価設計まで含めて導入する必要があると説明できる。
- **長文要約・議事録処理の設計**: 議事録、社内規程、問い合わせ履歴、CRMメモなどは、本文と最終指示の距離が品質に影響する前提でテンプレート化する。
- **Ollama運用ノウハウ**: Modelfile の `SYSTEM` / `TEMPLATE` / `PARAMETER num_ctx` は顧客ごとの標準動作を定義する入口になる。ただし、実際の順序や効き方はモデルとテンプレート依存なので、検証ログを残す。
- **有償診断・レビュー面談の素材**: 「社内資料をAIに読ませても回答がぶれる」企業向けに、原因をモデル性能だけでなくコンテキスト設計として説明できる。
- **評価観点**: 文字数、要点の網羅性、用語の正確性、構造化出力、不要な補足の少なさをテストケース化し、複数配置で比較する。

## 採用しない要素

- 元動画の話者名、チャンネル名、固有アプリ名、個人的なデモ例、宣伝CTAは採用しない。
- 「小規模モデルは大規模モデルよりノイズが少ない」「繰り返せば大規模モデルに勝てる」といった一般化は、一次情報で確認しにくいため公開記事では断定しない。
- ChatGPTやGeminiによるスコア比較は、再現条件が不十分なため根拠として使わない。
- 「コンテキストファーストなら必ず精度が上がる」という断定は避け、用途・モデル・テンプレートに応じて検証する方針に変換する。
