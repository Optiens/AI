# 文字起こし処理フロー（記録 → 報告 → 記事化）

CEO から動画・対談・ウェビナー等の文字起こしテキストが共有された際に実行する標準フロー。
3 ステップ（メモリ記録／CEO への報告／ブログ記事化）を一貫して進める。

## 起動条件

CEO が以下のような形で文字起こしテキストを共有したとき、本スキルを自動的に発動する:
- 「以下は動画の文字起こしです」「文字起こしを共有します」等の前置き
- 数百〜数千語の対談・講演・インタビュー風のテキストの貼り付け
- 文字起こし共有後に「Optiens に有用な内容を記録」「記事化」の指示があるとき

CEO から明示的な指示がなくても、文字起こしであることが文脈から明らかなら本フローを進めて確認を取る。

---

## Step 1: 文字起こし内容の整理・分析

### 1-1. 全体把握
- 発信者・登場人物・テーマ・日付（推定可なら）を把握する
- 全体の論旨を 100〜200 字で要約する

### 1-2. Optiens への有益度を評価する観点
以下の観点で「Optiens にとって有益な情報」を抽出する。

| 観点 | 抽出対象 |
|---|---|
| **市場・業界トレンド** | AI 業界の最新動向、競合の動き、技術ロードマップ |
| **顧客理解** | 中小事業者の課題・悩み・判断軸 |
| **マーケティング・集客** | 集客導線、ブランディング、コミュニケーション設計 |
| **サービス設計** | 新しいサービスメニュー、料金構造、提供方法のヒント |
| **経営判断** | 採用・組織・財務・戦略への示唆 |
| **コンテンツ素材** | 引用できる発言、独自フレームワーク、数値・統計 |
| **リスク・注意点** | 採用すべきでない手法、避けるべき表現、規制動向 |

### 1-3. 取り入れない要素の判別

Optiens のブランドトーン（落ち着き・誠実・実務）と相反する要素は、**構造のみ借用し表現は採用しない**:
- 煽り語彙（「絶対」「人生変わる」「いま動かないと」）
- 確約表現（「必ず成功する」「○○日で稼げる」）
- 不安喚起マーケ（焦らせて判断を急がせる）
- 過剰な権威付け

## Step 2: メモリへの記録

### 2-1. メモリ種別の判断

| 観点 | メモリタイプ | 例 |
|---|---|---|
| 業界の動向・他社事例・動画の参照 | reference | 「○○氏の集客手法分析」「AI 業界 ○年 △月の発言整理」 |
| Optiens の方針決定に関わる気づき | feedback | 「○○トーンは採用しない」「△△を必ず実施する」 |
| 進行中プロジェクトへの影響 | project | 「□□戦略を Phase A2 で開始」 |

### 2-2. メモリファイル作成

`C:\Users\blueb\.claude\projects\c--workspace-optiens-hydroponics\memory\` 配下に Markdown を作成。

ファイル名規則:
- `reference_<英語スラッグ>.md`
- `feedback_<英語スラッグ>.md`
- `project_<英語スラッグ>.md`

冒頭にフロントマター:
```markdown
---
name: 日本語の名前
description: 1 行で内容を表す説明
type: reference | feedback | project
---
```

本文の構成:
- 出典（誰の発言か・動画タイトル・日付）
- 中核メッセージ（200 字以内）
- Optiens への適用ポイント（具体的に）
- 採用しない要素（あれば）
- 記事化候補（あれば）

### 2-3. MEMORY.md への索引追加

`C:\Users\blueb\.claude\projects\c--workspace-optiens-hydroponics\memory\MEMORY.md` の末尾に 1 行追加:

```
- [日本語タイトル](ファイル名.md) — 1 行の要約
```

## Step 4: ブログ記事化（CEO の承認後 or Auto モードでは自走）

### 4-1. write-blog-article スキルを呼び出す

記事化候補ごとに、`.claude/skills/write-blog-article.md` の手順に従って記事を書く。
特に以下を遵守:
- ファクトチェック必須化（`feedback_blog-factcheck-mandatory.md` 参照）
- `src/content/blog/_project-status.md` の禁止表現リストを確認
- カテゴリは `AI / business / technology / agriculture / welfare / social / announcement` のいずれか
- 数値・固有名詞は出典明記または範囲表現
- 煽り語彙・確約表現・確証バイアスを誘発する表現は使わない

### 4-2. ブログ候補リストへの追記

`C:\Users\blueb\.claude\projects\c--workspace-optiens-hydroponics\memory\project_blog-candidates.md` に該当記事を ✅ 付きで追加する（記事化前なら未マークで追加）。

### 4-3. コミット・プッシュ

[feedback_commit-push-autonomy.md](C:\Users\blueb\.claude\projects\c--workspace-optiens-hydroponics\memory\feedback_commit-push-autonomy.md) に従い、可逆な記事追加は CEO の承認待ちなしで `git add → commit → push origin main` まで実行する。

コミットメッセージ例:
```
feat(blog): 文字起こし分析から記事 N 本を執筆

- ファイル名 (#候補番号): 1 行要約
- ファイル名 (#候補番号): 1 行要約

出典: <動画/対談タイトル等>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

## 例外・注意事項

### 文字起こしが部分的・不明瞭な場合
全体の論旨が読み取れない場合は、Step 1 の段階で「文字起こしの欠落・不明瞭部分があるため、可能なら原文 URL や追加情報を提供してほしい」と CEO に確認する。

### 機密情報を含む場合
文字起こしに顧客名・個人情報・取引条件等の機密情報が含まれる場合は、メモリ記録時に**マスキングまたは抽象化**してから保存する。

### 競合事業者を直接批判するトーンになりそうな場合
他社・他者を否定的に扱う記事は Optiens のブランドトーンと相反する。「採用しない要素」として整理し、肯定的な学びだけを記事化する。

### Optiens の事業段階（Phase A1）と相反する記事
「導入実績がある」「○○件支援した」等の実績語が文字起こし元の文脈にあったとしても、Optiens の記事では使えない。**Optiens 自身の記事に翻訳する際に必ず弱化する**。
